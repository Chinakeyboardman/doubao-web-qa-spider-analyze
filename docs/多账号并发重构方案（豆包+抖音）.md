# 多账号并发重构方案（豆包+抖音）

## 核心原则

- 并发不再基于同账号多会话，而是**账号池并发**。
- 每个账号独立：登录态、浏览器 profile、任务分片、失败重试。
- 默认 `1` 路（兼容当前），可配置 `1-5` 路。

## 目标结果

- 豆包 Step1 不再成为串行瓶颈：可按账号池并发采集。
- 抖音采用独立账号 worker 池，每账号独立 `user-data-dir`。
- 异常（验证码、风控、网络）只影响单账号 worker，不阻塞全局。

## 设计要点

1. **账号池抽象（先豆包后抖音）**

- 在豆包采集侧引入 `account_slot`（如 `slot_01..slot_05`）。
- 每个 slot 独立 `state.json`（或 `state_slot_01.json`）与浏览器上下文。
- 手机号验证码接口由于“部分支持”，第一版采用：
  - 串行领取号码/验证码
  - 成功后进入账号池轮转复用

1. **豆包并发采集执行模型**

- `pipeline.step_collect` 增加 `collector_concurrency`（默认 1）。
- 使用 `asyncio.Semaphore` 控制 worker 数；每个 worker 绑定一个账号 slot。
- 失败隔离：单 slot 遇验证码/封控只熔断该 slot，其他 slot 继续。
- 质量守门继续保留：空 answer/0 链接直接 error，不置 done。

1. **抖音独立 worker 池**

- `douyin-scraper.js` 支持 `--account-slot` 或环境变量 `DOUYIN_ACCOUNT_SLOT`。
- 根据 slot 自动映射独立 `BROWSER_USER_DATA_DIR`（如 `.browser_state/douyin/slot_01`）。
- `douyin_video.py` 调用本地 scraper 时透传 slot（由调度层分配）。
- 若 slot 登录失效，输出 action-required（含 slot 编号），只暂停该 slot。

1. **CLI 与调度参数**

- `run/collect/run-until` 增加：
  - `--collector-concurrency`（默认 1，范围 1-5）
  - `--account-slots`（可选，默认自动 `slot_01..slot_N`）
- 保持现有 `--crawl-concurrency`、`--category-prefix`、`--limit`。

1. **运行监控与恢复**

- 新增按 slot 的监控日志：成功率、验证码次数、连续失败次数。
- 自动降级策略：某 slot 连续失败阈值触发熔断，任务回到 pending 队列。
- 断点恢复：重启后按 slot 状态继续消费，不重跑已 done 数据。

## 执行顺序（建议）

1. 豆包账号池并发（slot + concurrency + 日志）
2. 抖音 slot 化 worker 池
3. `run-until` 接入 slot 监控与熔断
4. 小流量压测（1→2→3 路）再放开到 5 路上限

## 验收标准

- 同配置下，相比单路：Step1 吞吐提升明显（2-3 路线性接近）。
- 任一账号异常不会导致全局停摆。
- 不再出现 `done` 但 `qa_answer` 空值。
- 跑批中断后可无损续跑（无大面积重算）。

## 风险与缓解

- 手机号接口吞吐不足：先做“串行领号 + 池复用”，避免并发争抢。
- 验证码高频触发：slot 熔断 + 人工补登录后恢复。
- 抖音风控：每 slot 限速、独立 profile、失败隔离。
