# Q0320 起回答无参考资料（引用）— 原因与处理

> **重要**：单条 query **不能**用「采集结果 0 条链接」反推采集器坏了。例如 **`机械键盘轴体有什么区别红轴青轴茶轴怎么选`**（对应库内如 Q0320）在豆包侧**可能本身就没有「参考资料」**，属于正常业务结果。验证采集器应选用 **人工确认页面上确有可点外链 / 参考列表** 的 query，或对比 **同类型 query 在旧批次有引用、新批次全无** 的统计规律。

## 结论（已核实数据）

- **现象**：从 **Q0320** 起，大量 `qa_answer.has_citation=0`、`citation_count=0`，`raw_data` 里 `"links": []`，且 **`qa_link` 中对应 query 行数为 0**。
- **对比**：**Q0319 及以前**（例如 Q0318、Q0319）同库中 **`has_citation=1`**，且 `qa_link` 条数与 `citation_count` 一致。
- **时间线（示例）**  
  - Q0318/Q0319：`qa_query.updated_at` 约在 **2026-03-20 14:50–14:52** 采集完成，引用正常。  
  - Q0320 起：约在 **2026-03-20 16:26** 之后批量采集，**引用全部为空**。

因此问题不在「入库少写字段」，而在 **网页采集阶段没有解析到任何外链**（`integration/doubao_web_collector.py` 里 `deep_links` + `inline_links` 为空）。

数据来源均为 **`raw_data.source = web_ui`**，排除「误用 API 采集（无链接）」这一分支。

## 根因说明（推断 + 工程上可验证）

1. **未进入「思考 / 深度思考」模式**（最可能）  
   采集前会调用 `_switch_to_think_mode()`。历史上依赖 **`快速` → 点 `思考`**。豆包若改版（例如改为 **「自动/极速」**、或主按钮改为 **「深度思考」**），旧选择器会失效，脚本在 **非联网思考模式** 下生成回答，**页面上本身就没有「参考资料」结构**，则所有 DOM 抽取逻辑都会得到 0 条链接。

2. **正文已结束，但参考资料区块尚未渲染完**  
   `_wait_until_done()` 在回答正文连续一段时间不变即结束；若新版 UI 中 **参考资料晚于正文出现**，可能在抽取时 DOM 仍为空。可通过 **延长正文稳定后的等待** 缓解。

3. **页面 DOM / class 变更**  
   `_extract_deep_thinking_links_js` / `_get_answer_inline_links` 依赖 class 与结构；大改版时可能需再调选择器（一般弱于 1）。

## 代码侧已做改进（`doubao_web_collector.py`）

- **更稳健的思考模式切换**：支持多种入口文案（如「快速/极速/自动」）与选项（「深度思考」「思考」），并记录 **切换失败** 的明确日志。
- **新版参考资料入口 v3**：优先点击 **`[data-testid="search-reference-ui-v3"]`**（文案如「参考 79 篇资料」），失败时用 **`[class*="entry-btn-v3"]`** + 文案匹配；展开后再跑统一 DOM 抽取。
- **可配置等待**：环境变量 **`DOUBAO_WEB_POST_ANSWER_SETTLE_MS`**（默认 8000ms），在正文就绪后 **再等待** 一段时间再抽链接。
- **可选二次抽取**：**`DOUBAO_WEB_CITATION_RETRY_MS`**（默认 0）：首次 0 条时等待后再抽一轮（用于排查「DOM 晚到」）。

## 建议你本地做的验证

1. **有头浏览器**：`python integration/run.py web-login --manual`，手动发一条 query，确认界面里模式是否为 **思考/深度思考**，参考资料是否出现。  
2. **重采样本**：对个别 `query_id` 使用 `recollect` / 网页采集单条，观察日志是否出现 **`未能切换到「思考」模式`** 或 **`No citation links captured`**。  
3. **调参**：若确认是「DOM 晚到」，可适当增大 `DOUBAO_WEB_POST_ANSWER_SETTLE_MS` 或设 `DOUBAO_WEB_CITATION_RETRY_MS=12000` 试跑。

## 历史数据补救

对已落库、无 `qa_link` 的 query，需要 **重新网页采集**（或后续若支持从页面导出/API 再写 `qa_link`）。仅改库字段无法恢复真实 URL。

## 实测：`recollect-web-only --query-ids Q0320`（**不宜作为「采集器失效」判据**）

- **Q0320 的 query 文案**（见上）在豆包上**可能本来就不提供参考资料**，因此用该条 **0 条 `qa_link`** 不能说明 DOM 解析失败。
- 若仅作技术侧观察：日志里 **已成功切换到「思考」**；重采后仍为 0 条链接时，应结合 **页面是否真有参考区** 再判断；若页面上就没有外链，则采集结果为 0 是 **预期**。
- 代码侧增强（统一抽取、展开面板、data-* 等）仍适用于 **确有引用 UI** 时的兼容；若仍无引用，需对照 `after_reference_panel.png` 区分「产品无引用」与「有引用但 DOM 未暴露」。
